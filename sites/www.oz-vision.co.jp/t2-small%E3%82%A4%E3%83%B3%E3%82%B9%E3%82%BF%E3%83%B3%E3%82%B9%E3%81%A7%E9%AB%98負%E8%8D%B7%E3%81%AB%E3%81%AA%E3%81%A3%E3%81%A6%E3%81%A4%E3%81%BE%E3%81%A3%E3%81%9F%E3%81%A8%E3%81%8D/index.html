t2.smallインスタンスで高負荷になってつまったとき

こんにちは。
制作開発2年目ソラです。

この記事は、webエンジニアの初心者が話していることだと頭に入れながら読んでいただきたい。
初心者が読んだときに、少しでも役に立ってくれると嬉しいと思って書いています。

新規webサイトを運営したときに、t2.smallを使っていたので、
その時に詰まったときのことをまとめます。

前提条件
・パッケージを購入しているため、アプリケーションがどうなっているか把握しきれていない
・新規webサイトで、お金をかける方向に倒したくない
・railsもnginxもawsも初めて触るエンジニア

使用環境
nginx + passenger
ruby on rails
aws アプリケーション側 t2.small
起きたこと
・disk容量が100%になり、キャッシュが残せなくなりレンダリングができなくなった
・CPUが閾値を超えタイムアウトするようになった
・disk容量が100%になった時の調査方法と対策

調査方法
・ディスク容量を食いそうな箇所を検索
└webサーバにアプリケーション直下のcacheを書き込んでいるのを発見
└webサーバにlogを書き込んでいるのを発見
対策
・cronで、5時間おきにcache clearする処理を作成（cache clearの処理自体はパッケージに入っているものを使用）
・access_logとerror_logをweeklyでlog rotateして手動で、バックアップを取り削除するようにした
・railsのproduction_logをlog rotateするようにした。
（config.logger = Logger.new(“log/production.log”, 5, 10 * 1024 * 1024)）
・CPUが閾値を超えたときの調査方法と対策
調査方法
・タイムアウトしたらわかるように「nagios」を使った監視
・タイムアウトした時間の、access_logを確認
どの処理が待ち状態になるか調査するため、/opt/nginx/conf/nginx.confの log_formatに「$request_times」を追加

log_format main ‘$remote_addr – $remote_user [$time_local] “$request” ‘
‘$status $body_bytes_sent “$http_referer” ‘
‘”$http_user_agent” “$http_x_forwarded_for” $request_time';

・チェックリストを作成し、一つずつ原因だと思う箇所を潰していく（githubのissueを使って管理）
調査内容もすべて、issueに残して管理していた。

・アプリケーション側を調べつくしたのちに、t2.smallについて調査
http://techblog.heartrails.com/2011/01/rails-nginx.html
http://qiita.com/fnifni/items/4395801eba4d7d5fdd4f

対策
・調査した結果アプリケーションに問題がないと判断し、インスタンスサイズを変更
学んだこと、大事だと思ったこと
・稼働開始前に、log rotateの設定をいれ、ディスク容量を圧迫しないようにする
・いつでも調査できるように監視もしくは、アラートを設定
・キャッシュを仕込んでいる際は、キャッシュしている箇所と容量に気を付けて、適切にlog rotateする
・調査できるように、log_formatの設定をする
・月間PV数10万ぐらいでt2.smallを使用している場合、cpu利用率20%を超えるケースがある場合、インスタンスサイズの見直しを考えたほうが良い
（これはあくまで個人的な感覚です。）
・表示できない状態が頻発する場合は、調査は後にして、インスタンスタイプを切り替えから実施することを感が他方が良い
今後もわからないこと、知らないことに果敢に挑戦し続けようと思います！
